# 校园外卖信息聚合平台 — 项目结构说明

**文档类型**：项目结构 / 分层设计 / DTO/VO 约定  
**适用范围**：外卖聚合模块后端（Spring Boot）、后台前端（Vue3）、小程序端（微信原生）

---

## 1. 后端整体分层

后端推荐采用典型的“分层 + 领域模块”结构：

- **接口层（Controller 层）**
  - 职责：接收 HTTP 请求、参数校验、鉴权、组装/返回 VO。
  - 命名：`*Controller`，路径前缀 `/api/mp`、`/api/admin` 等。
- **应用层（Service 层）**
  - 职责：封装业务用例（用例级服务），处理事务、组合多个领域服务与仓储。
  - 命名：`*Service` 接口 + `*ServiceImpl` 实现。
- **领域层（Domain 层）**
  - 职责：核心领域模型、领域对象的业务规则（如评分计算、推荐规则等）。
  - 包含：`domain` 模型类、领域服务（如 `RecommendDomainService`）。
- **基础设施层（Infra/Repository 层）**
  - 职责：与数据库、缓存、MQ、第三方服务的交互。
  - 包含：MyBatis-Plus Entity、Mapper、Repository 封装、Redis 访问等。
- **通用支持层（Common/Shared）**
  - 职责：统一返回体、错误码常量、异常、工具类、配置等。

## 2. Maven 模块与包结构建议

### 2.1 Maven 模块（可选）

若仅外卖模块，初期可以单模块 `waimai-backend`；若后续扩展较多，可拆分为：

- `waimai-backend`：后端主应用
- `waimai-backend-common`：公共通用模块（可选）

> 初期可以全部放在单一 Spring Boot 工程中，通过包结构划分领域。

## 3. DTO / VO / Entity 命名与约定

### 3.1 概念区分

- **Entity（实体）**
  - 对应数据库表结构，如 `ShopEntity`、`ShopRatingEntity`。
  - 仅在 `repository`/`mapper` 层使用，不直接暴露给外部接口。
- **DTO（Data Transfer Object）**
  - 主要指**请求体**或服务内部传输对象，如 `CreateShopRequest`、`UpdateShopStatusRequest`。
  - 用于 Controller 层接收参数、Service 层调用时传递数据。
- **VO / Response（视图对象 / 响应体）**
  - 用于对外返回的对象，如 `ShopListItemVO`、`ShopDetailVO`、`DailyRecommendVO`。
  - 字段只包含前端需要展示的内容，不包含内部实现细节（如 `deleted`）。

### 3.2 命名规范

- 请求体：
  - 后台管理：`XXXRequest`，如 `CreateShopRequest`、`UpdateCategoryRequest`。
  - 小程序端：可根据场景命名，如 `MpShopRatingRequest`。
- 响应体：
  - 单对象：`XXXVO`，如 `ShopDetailVO`、`AdminLoginVO`。
  - 列表项：`XXXListItemVO`，如 `ShopListItemVO`、`FavoriteShopItemVO`。
  - 分页封装：统一使用 `PageResult<T>`（放在 `common.api` 包）。
- Entity：
  - 与表名对应，如 `ShopEntity` 对应 `shop` 表，`ShopFavoriteEntity` 对应 `shop_favorite` 表。

### 3.3 典型示例（文字说明）

- 小程序商家列表接口 `GET /api/mp/shops`
  - Controller：
    - 入参：分页参数 + 筛选条件（可直接用 `@RequestParam` 或封装为 `ShopListQuery` DTO）。
    - 出参：`PageResult<ShopListItemVO>`。
- 商家详情接口 `GET /api/mp/shops/{id}`
  - 出参：`ShopDetailVO`，包含平台链接列表、评分统计、收藏状态等。
- 后台新增商家接口 `POST /api/admin/shops`
  - 入参：`CreateShopRequest`。
  - 出参：`IdVO`（仅返回新建 ID）。

## 4. 统一返回与分页封装

### 4.1 统一返回模型

在 `common.api` 包中定义统一响应类（示意）：

- `ApiResponse<T>`
  - `code`：int，错误码（参见 `docs/错误码设计.md`）
  - `message`：String，错误信息或“OK”
  - `data`：T，业务数据

Controller 层统一返回 `ApiResponse<T>`，通过静态方法如 `ApiResponse.success(data)`、`ApiResponse.error(code, message)` 生成。

### 4.2 分页封装

- `PageResult<T>`
  - `total`：long
  - `page`：int
  - `pageSize`：int
  - `list`：List<T>

> 与《详细设计-API 接口设计》中对分页字段的约定保持一致。

## 5. 前端项目结构简要建议

### 5.1 后台管理前端（Vue3 + Vite）

推荐目录（简化示例）：

```text
src
├── api           # Axios 封装 & 各模块接口定义
├── views         # 页面级组件（ShopList、ShopEdit、RecommendConfig 等）
├── components    # 复用组件（表格、表单等）
├── router        # 路由配置
├── store         # Pinia 状态（用户信息、菜单等）
├── utils         # 工具方法
└── types         # TypeScript 类型定义（与后端 VO 对应）
```

### 5.2 小程序端

按功能拆分页面与公共模块，例如：

```text
miniprogram
├── pages
│   ├── waimai-index     # 外卖首页（每日推荐+列表入口）
│   ├── waimai-list      # 商家列表
│   ├── waimai-detail    # 商家详情（评分/收藏/跳转）
│   └── waimai-favorite  # 我的收藏
├── utils
│   ├── request.js       # 封装 wx.request，统一带 token 与错误处理
│   └── auth.js          # 登录 & token 处理
└── config
    └── env.js           # 后端接口基础地址等
```

## 6. 后续演进建议

- 若项目演进为“外卖 + 二手交易”多模块，可：
  - 在包结构中增加 `trade.*`、`secondhand.*` 等新领域包。
  - 提取更通用的 `user`/`account` 模块到公共层。
- 当团队人数增加时：
  - 可将每个模块拆成独立 Maven 子模块（如 `waimai-module-shop`）。
  - 增加更完备的领域层建模（值对象、聚合根等）。
