## 2. 后端（Spring Boot）

### 2.1 核心版本（已确认）

| 技术            | 版本/范围 | 备注                                          |
| --------------- | --------- | --------------------------------------------- |
| JDK             | **21**    | 当前 LTS 版本，推荐新项目使用                 |
| Spring Boot     | **3.2.x** | 3.2 系列稳定成熟，与 JDK 21 兼容              |
| Spring Security | 同 Spring | 管理端鉴权、小程序 openid 校验                |
| MyBatis         | 3.0.x     | 与 MyBatis-Plus 兼容的版本                    |
| MyBatis-Plus    | **3.5.x** | 单表 CRUD、分页、条件构造器，减少重复样板代码 |

### 2.2 数据库与连接池

| 技术     | 推荐         | 说明                                  |
| -------- | ------------ | ------------------------------------- |
| MySQL    | **8.0.x**    | 使用 utf8mb4、常用 8.0 语法与窗口函数 |
| 连接池   | **HikariCP** | Spring Boot 默认，无需额外依赖        |
| 分库分表 | 不做         | 单校单校区数据量无需分库分表          |

### 2.3 Redis 使用场景建议

| 场景              | 是否建议使用 | 说明                                                          |
| ----------------- | ------------ | ------------------------------------------------------------- |
| 每日推荐结果缓存  | ✓ 推荐       | 按自然日缓存「今日推荐」商家 ID 或简要信息，减少重复计算      |
| 商家列表缓存      | 可选         | 若列表 QPS 不高可先不缓存，直接查库                           |
| 会话 / Token 存储 | 可选         | 若采用 JWT 无状态则可不存 Redis；若用 refreshToken 可存 Redis |
| 限流 / 防刷       | 可选         | 登录、评分、收藏等关键接口可做简单限流                        |
| 分布式锁          | 按需         | 多实例部署时如需「当日推荐只算一次」可考虑                    |

### 2.4 消息队列（MQ）选型与使用场景

| 项目        | 选型             | 说明                                                                                                           |
| ----------- | ---------------- | -------------------------------------------------------------------------------------------------------------- |
| **MQ 产品** | **RabbitMQ 3.x** | 中小规模、运维简单、社区成熟；与 Spring AMQP 集成好。RocketMQ/Kafka 可在后续有高吞吐、日志审计等需求再考虑引入 |
| 使用场景    | 一期可简化       | 见下表                                                                                                         |

| 场景               | 是否建议 MQ | 说明                                                    |
| ------------------ | ----------- | ------------------------------------------------------- |
| 操作日志异步落库   | 可选        | 可先同步写库，后期再改为发 MQ 异步写，减轻主链路压力    |
| 申请通过/驳回通知  | 可选        | 需求说明书未强制站内信/推送，可二期再做，届时用 MQ 解耦 |
| 定时任务触发的统计 | 一般不需要  | 用 Spring 自带 `@Scheduled` 即可                        |

**结论**：技术栈上**确认使用 RabbitMQ**，一期可只做**基础接入（连接、一个简单队列）**，具体业务异步化在二期按需加。

### 2.5 其他后端细节

| 项目         | 建议                                                                                                              |
| ------------ | ----------------------------------------------------------------------------------------------------------------- |
| 构建工具     | **Maven** 或 **Gradle**（二选一，团队统一即可）                                                                   |
| 接口风格     | **RESTful**，统一返回封装（code、data、message）                                                                  |
| 鉴权         | **JWT**（accessToken + 可选 refreshToken）；管理端与小程序端用同一后端时可通过 token 中的 claims 或请求头区分身份 |
| 接口文档     | **SpringDoc OpenAPI 3**（Swagger 3）或 **Knife4j**，便于前后端联调                                                |
| 参数校验     | **Jakarta Validation**（@NotNull、@Size 等）                                                                      |
| 全局异常处理 | `@ControllerAdvice` + 统一错误码                                                                                  |
| 日志         | **SLF4J + Logback**，按需输出 JSON 便于采集                                                                       |
| 配置         | **application.yml** + **spring.profiles.active**（dev/test/prod）；敏感信息用环境变量或配置中心                   |

## 3. 后台管理端（Vue 3）

### 3.1 核心版本与工具

| 技术        | 推荐版本/选型       | 说明                                            |
| ----------- | ------------------- | ----------------------------------------------- |
| Node        | **18.x / 20.x LTS** | 与 Vite、Vue 3 兼容                             |
| Vue         | **3.4+**            | Composition API + `<script setup>`              |
| 构建工具    | **Vite 5.x**        | 开发体验好、构建快                              |
| 语言        | **TypeScript**      | 推荐，接口类型与可维护性更好                    |
| 路由        | **Vue Router 4.x**  | 仅管理员一种角色，路由可简单配置                |
| 状态管理    | **Pinia**           | Vue 3 官方推荐，替代 Vuex；可存用户信息、菜单等 |
| HTTP 客户端 | **Axios**           | 封装 baseURL、请求/响应拦截器、统一错误提示     |

### 3.2 UI 组件库

| 选型           | 说明                                                             |
| -------------- | ---------------------------------------------------------------- |
| Ant Design Vue | 已选为后台主 UI 库，遵循 Ant Design 设计规范，组件丰富、文档完善 |

### 3.3 其他前端细节

| 项目       | 建议                                                                               |
| ---------- | ---------------------------------------------------------------------------------- |
| 样式       | **CSS 变量** + **Scss** 或 **Less**，与 Element Plus 主题变量配合                  |
| 权限控制   | 路由守卫根据 token 判断登录；若仅一种管理员角色可不做细粒度按钮权限                |
| 环境与代理 | `.env.development` 中 `VITE_API_BASE_URL` 指向后端；开发阶段用 Vite proxy 解决跨域 |
| 包管理     | **pnpm** 或 **npm**，建议锁定 lockfile                                             |

## 4. 小程序端

### 4.1 开发方式（已确认）

| 方案         | 优点                           | 缺点 / 注意                      |
| ------------ | ------------------------------ | -------------------------------- |
| **微信原生** | 无额外运行时、体积小、文档直接 | 仅支持微信，若以后要做多端需重写 |

**结论**：本项目小程序端采用 **微信原生** 实现，不引入 uni-app。

### 4.2 与后端协作

| 项目 | 建议                                                                                          |
| ---- | --------------------------------------------------------------------------------------------- |
| 请求 | `wx.request` 或封装成统一 request（baseURL、header 里带 token）                               |
| 登录 | 微信 `wx.login` 取 code，后端用 code 换 openid/session_key，再下发自定义 token（JWT）给小程序 |
| 鉴权 | 所有请求 header 携带 token；后端校验失败返回 401，小程序跳转登录页或重新静默登录              |

## 5. 基础设施与部署

### 5.1 存储与中间件

| 类型     | 选型           | 说明                                                   |
| -------- | -------------- | ------------------------------------------------------ |
| 对象存储 | **阿里云 OSS** | 商家封面图等静态资源；结合 CDN，满足小程序图片访问需求 |
| 数据库   | MySQL 8.x      | 单机或云 RDS                                           |
| Redis    | 单机或云 Redis | 开发可本机 Docker                                      |
| RabbitMQ | 单机或云版     | 开发可本机 Docker                                      |

### 5.2 部署方式（建议一期）

| 项目         | 建议                                                                                    |
| ------------ | --------------------------------------------------------------------------------------- |
| 部署形态     | **单机** 或 **单台云服务器**；后端、管理端静态资源、Redis、RabbitMQ、MySQL 可同机或拆分 |
| 后端         | 打 jar 用 `java -jar` 或 systemd 管理；或 Docker 单容器                                 |
| 管理端       | `npm run build` 后 Nginx 托管静态资源；或与后端同一 Nginx 反向代理                      |
| 域名与 HTTPS | 必须 HTTPS；微信小程序要求服务器域名备案与 HTTPS                                        |
| 配置         | 环境变量区分 dev/test/prod；数据库、Redis、MQ 等连接信息不写死                          |
